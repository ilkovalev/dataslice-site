# Design Review — DataSlice

Аудит **дизайна**: работает ли дизайн на свои цели, а не «влоб»-чтение кода.
Контент (понятность текста, тон) — отдельный скилл `content-review`, здесь НЕ рассматривается.

**Главный вопрос аудита:** работает ли текущий дизайн на свои цели — понятность для новичка
и путь к подписке на Telegram-канал — или его надо менять, вплоть до полной переделки.
Вывод «переосмыслить/переделать» — легитимный исход, не провал.

Референсы: `anthropics/skills` frontend-design (не AI-дефолт ли), `dataviz` (качество графиков).
Эталон намерения — `BRAND.md` (для кого сайт, цель = подписка).

## Когда применять
Дизайн-ревью или полный дизайн-аудит страниц (`/stats`, `/metrics`, `/glossary`) после
правок UI/виджетов, либо по запросу «оцени дизайн».

## Пайплайн

### 1. Захват (оба вьюпорта, все страницы — не выборка)
```
node .claude/skills/design-review/scripts/capture.mjs <slug>       # скрины + metrics.json
node .claude/skills/design-review/scripts/mobile-montage.mjs        # монтаж виджетов на мобильном
```
`capture` отдаёт `reports/shots/<slug>/`: скриншоты `mobile(390)/tablet/desktop` × RU/EN +
`metrics.json` (шрифты, контраст, тап-зоны, alt/aria, overflow, line-height прозы `tightProse`).
Для полного аудита — собрать монтажи ВСЕХ уроков на мобильном И полноширинном десктопе (1280px+),
судить по всем, не по выборке.

### 2. Аудит по 5 дизайн-линзам + 6 аспектов
Прогнать линзы (`rules/*.md`) по рендерам и метрикам. По каждому аспекту вынести вердикт
[оставить / доработать / переосмыслить] с доводами от лица пользователя:
1. Визуальный язык — отличителен/уместен или templated/устарел (frontend-design: не AI-дефолт).
2. Иерархия и читаемость — куда падает глаз, совпадает ли с главным.
3. Смысл и работа каждого ТИПА блока — что даёт, нет ли лишних/пустых/дублирующих.
4. Интерактив как инструмент понимания, не украшение (в т.ч. начальное состояние виджетов).
5. Путь к целевому действию (подписка).
6. Целостность и консистентность системы.

### 3. Аудит-отчёт
`reports/Дизайн-аудит_<дата>.md`: вердикт по сайту в целом + по каждому из 6 аспектов с
доводами + приоритизированные рекомендации + (если «переосмыслить») направление редизайна.
ОТДЕЛЬНЫМ списком, не как суть — гигиенические баги (см. severity).

## Процессные требования (уроки прошлых прогонов — соблюдать!)
- **Оба вьюпорта.** Каждую находку и правку проверять на мобильном 390px И полноширинном
  десктопе 1280px+. SVG-текст с узким viewBox РАЗДУВАЕТСЯ на десктопе — не чинить fontSize наскоком.
- **Сверять с рендером.** Инструментальная находка — кандидат, не приговор. Артефакты
  (KaTeX-формулы ≈17px; полупрозрачные фоны `bg-*/NN` в контраст-детекторе) — НЕ баги.
- **Не по выборке.** «Однородно, значит выборка ок» — ложная логика; смотреть все.

## 5 дизайн-линз
| Линза | Файл | Вход |
|---|---|---|
| Визуальный дизайн | `visual-design.md` | скрин + метрики (шрифты, ритм, line-height) |
| Качество графиков | `dataviz.md` | скрины состояний + контраст/оси |
| Интерактив | `interactivity.md` | Playwright-действия до/после |
| UX / смысл блоков | `ux-blocks.md` | скрин всей страницы + цель `BRAND.md` |
| Доступность | `accessibility.md` | метрики |

## Severity (для гигиенического списка)
- **blocker** — контраст ниже AA на основном тексте, интерактив не работает на мобильном, overflow ломает страницу.
- **важное** — заметно вредит, но страница юзабельна.
- **полиш** — ритм отступов, мелкие типо-выбросы.
